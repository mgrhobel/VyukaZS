Chapter 1

Introduction
Often the decision to sell a software on the global market is a requirement added late in
the software development process. Suddenly, a number of problems introduced by cultural
diversity arise. The impact of this is wider than expected. Not only different languages
have to be supported, but also symbols and the arrangement of dialog components has to be
changed. In the following the term language is used to refer to all of these differences.
This report explains the main difficulties when dealing with these issues. Simple approaches
like Resource Files are investigated and commercial solutions presented.

1.1

Terms & Definitions

Globalization
The process of Globalization of a software is usually divided in the terms Internationalization and Localization. For the last two terms abbreviations are widely used: i18n and L10n.
The abbreviations consist of the first and last letter of the word and the number of omitted
letters in between of them. The i is lower case to avoid the ambiguity with 1.
Sometimes Globalization is used as a synonym for Internationalization and vice versa.[ea98]
Outside the context of software development Globalization is used to reflect the intention of
targeting the global market.

1

The Localization Industry Standards Association (LISA) defines the term as follows:
We use globalization with a very specific meaning, usually to mean the process
of adapting products for international marketing and distribution, including all
of the business decisions needed to reach those markets.[Lom03]
Internationalization (i18n)
Basically, supporting different languages, etc. is achievable in two major ways. Technically
the easiest approach is to have multiple development branches of the software project, one
separate branch for each supported language/culture. However, this makes maintenance and
improvements difficult since changes have to be made to each branch individually. For small
projects this might be feasible. Large projects on the other hand can not afford the additional
burden implied.
An alternative is to make the application flexible and restructure it so that culture depended
information is extracted from data instead of being embedded in the code directly. Then,
supporting an additional language is a matter of translating the data files. Since there is one
development branch only, the ability of bug fixing and further development is preserved.
Preparing an application in such a way is called Internationalization, the resulting program is
an International Application. Usually, the application contains a build-in data set containing
the interface for the native language of the application. This eases development and serves
the purpose of a fall-back in case the user’s language is not available.
Localization (L10n)
Once the application is sufficiently prepared, a new language is added by providing a
translated resource file. The process of building such a resource and adding culture-specific
details to the application is referred to as Localization.
Translation
This term is used specifically for the process of translating texts to another language.
Localization includes the translation of icons, videos, etc. in addition. The distinction is
made because usually the actual Translation is provided by people outside the software
development field.

2

GILT industry
A whole industry supplies tools and services for the development of international software.
Its name is GILT: Globalization, Internationalization, Localization and Translation.[Lom03]
Other terms are introduced as needed.

1.2

Consequences of Localized Versions

No matter how localized versions are created, the support implies some overhead, even for
translating data only. Development is more complex, another source of errors is introduced.
In general, it is a good idea to support different languages when the product is sold abroad.
Accessibility is improved. More users are enabled to benefit of the application. People not
speaking the native language of the product become potential customers. Other people do
not depend so much on localized versions, but might consider the support as an indication
of quality.
Certain types of applications depend more on localization than others. Consider writing
a letter using a word processor. The spell checking feature is completely unusable if the
dictionary for the used language is unavailable. An other example is a accounting software
unaware of the local currency and tax policies.

User’s Perspective:
Differences among Countries
As mentioned in the introduction there is more to consider than just the different languages
when globalizing an application.

2.1

Language & Text Display

Different languages are used in different countries. This is both the most obvious and the
most time-consuming factor of Localization. Each foreign language supported requires the
translation of all texts displayed by the application.
Time can be saved by first defining the major language groups to be supported. The text
is translated from the native language of the application as the starting point. Additional
translations based on the groups are performed to account for local differences. Generally, translations are done by representatives of the target user population. An example
of a language group is English with the sub-languages American English and British
English.[DD01]
In some areas of the world multiple languages are spoken, like in the border regions of France.
Here, a multi-user application should support switching to a different interface language onthe-fly. Furthermore, depending on the language the same information usually requires a
different amount of screen space. Thus, it can be necessary to adjust the layout of the in4

terface to provides sufficient space. Some languages are read and written from right to left
instead of left to right. Usually, this also affects the placement of buttons, control elements
and labels especially. In most cases, users reading from right to left expect the labels to
appear on the right side of the component they belong to!

2.2

Contact Information

The address format varies across countries. Europeans will have difficulties providing data
for all fields of American forms. Not every country is divided in states, for example. When
the software ensures non-empty fields users are forced to enter dummy data. This leads to
confusion and frustration.
The American 3-3-4 format of telephone numbers is not used world-wide. The number might
be longer than 10 digits. Furthermore, this format omits the international country code.
A good solution is to let the user select the country for the contact information to be entered.
Then the user-interface can be adjusted.[DD01]

2.3

Time & Date

The display of times and dates differs not only in the delimiters used. The order of the numbers varies as well. American users expect dates to be presented in the form of mm/dd/yyyy,
whereas Germans assume dd.mm.yyyy. Also, the format is a property of the country the
user lives in, not of the language spoken.[DD01]

2.4

Units

At least for scientific applications the metric system is sufficiently wide accepted to be used
across all localized versions. For convenience however, the application should allow entering
data using the unit the user chooses. That way, the user does not need to convert the unit
herself. To avoid ambiguities the unit should always be displayed along with values.[DD01]

5

2.5

Currency

Generally, currencies are not convertible since the conversion factor changes over time. Also,
currencies symbols are prepended or appended. Many applications require the system to
handle multiple currencies at the same time.

2.6

Icons & Symbols

Most images used in tool bars, menu bars and dialog boxes relate to the real world in the form
of a user-interface metaphor. The most common example is the postbox icon of electronic
mail programs. The U.S. postbox with the flag is quite different from the European boxes
with slots. Because of the binding to objects specific to a country all images have to be
reviewed. Appropriate replacements are provided where necessary.[DD01]

2.7

Video, Animations & Multimedia Content

The same process is required for videos, animations and other types of multimedia content.
Special care must be taken where people are displayed. Cultural differences not only includes
their appearance (gender, race, outfit) but their behavior as well. The means of nonverbal
communication like gestures and mimics1 can be quite different, even opposing.[MZ]
Crossing the hands behind the head is an example of a problematic gesture. While in western
countries this is a symbol of relaxing, in Russia it is considered a sign of ill manners.[RC]

1 gestures are a part of the body language, mimics are facial expressions.

Experience showed late Internationalization of software is expensive in terms of both
complexity and money.[ea98] It is just not feasible to translate the whole interface defined
by source code. Clearly, culture specific elements have to be separated from the code of the
application. This requires to think about a clean way of separation before implementing the
application.
So far the focus has been the user interface. Considering the pieces making up a modern
application it is a small part though. Culture has an influence on the data processed as
well. A good example is an application supporting the employees of a mail order company.
The company has a single head quarter and no stores abroad. Thus, a translation of the
interface does not appear to be necessary. What if articles are imported from a foreign
merchant? – Since the merchant is abroad its contact information require the application to
handle international data. Also, business may be expanded using a web interface allowing
foreign customers to place orders. This example shows that even applications with a local
user interface benefit from internationalizing the data model.

4.2

Resource Files & Keys

All data files an application needs in order to run properly are resource files. An international
application comes with a set of culture-specific resources. When switching from one language
to another the application simply uses a different set of resource files.
Usually translations such as texts are grouped logically together. They are easily accessed
using resource keys. A simple approach is to use a text file:

#resource file for the German language support
#notation: key = value
Hello world! = Hallo, Welt!
How are you today? = Wie geht es dir heute?
Thanks = Danke
You are welcome = Bitte schoen
#<--problem with special character! (oe)
Cheers! = Prost!
Here, the native language texts are used as keys directly. This eases the process of
translation and ensures the program works properly even if no translation is accessible. As
the example shows this approach is limited in representing special characters.

4.3

Character Sets

The above problem steams from the fact that different cultures use different accent characters
and other symbols. A special case are the Ashien languages where one-’letter’ symbols form
a word. A comprehensive alphabet containing all characters of all languages that exist is far
too huge to be handled by the usual 8-bit character representation. To solve this problem
a number of character used by a language are combined to form a character set.1 Other
common terms for character set are code set or code page.
Any Personal Computer is configured to use the character set local to its country. The
character set is part of the locale. When developing applications it is often overlooked that
the interpretation of character data is always dependent on the associated character set.
Since the system provides a reasonable default none-international applications usually do
not have to deal with it.
Developing international applications requires the correct use of character sets! Otherwise
1 along with symbols and control characters.

10

data using a foreign character set, that is not the system default, is not interpreted correctly
and errors will result. A solution is to convert character data from external sources to an
internal independent representation. A common representation is Unicode which uses 32-bit
per character to accommodate a general purpose alphabet. The term wide-character is often
used to refer to sets using more than 8 bits per character.
Since Unicode is space-consuming the standards UTF8 and UTF16 were derived. Here the
most common characters of the Unicode alphabet are represented by 8 or 16 bit respectively.
The remaining characters are accessible by means of an escape character followed by a code.
That way less space is ’waisted’ while maintaining the full Unicode alphabet.
An alternative to Unicode often mentioned are Multibyte Character Sets (MBCS) which are
commonly used on the Microsoft Windows platform.2 MBCS is a hybrid set using 8 or 16
bits to represent a character. In essence the ASCII characters are used. A range of codes
is reserved as lead-bytes which indicate the use of another code page for the byte following
it. That way Ashien language alphabets fit in a Multibyte character set at least. However,
MBCS still requires to select the proper MBCS code page. One reason to favor MBCS is
that existing applications are easier to adopt to it.
A good example of international software that has to handle text data using different
character sets are XML-Parsers. This type of software has to handle different sets even at
the same time since each XML document may specify the character set used freely. Most
XML-Parsers translate the data to UTF-16 Unicode to process it.

4.4

XML data exchange

As mentioned above XML technology eases the handling of different character sets. The
problem arises when data is exchanged, by transferring a file for example. Specifically this
means the import of data from or the export of data to another application. In addition,
international applications have to handle the im- and export of data by the same application
in another localized version!
XML solves both issues because information about the character set is embedded in the
document and the middleware handles proper conversions.

5.1

SDLinsight

SDLinsight of SDL International (http://www.sdlintl.com/) is a localization tool for Microsoft Windows user interfaces. In order to use the tool proper Internationalization of the
application is necessary.
In the context of the product this means that all texts, dialog-templates, icons and images
have to be loaded from the resource section of Windows binary files. This is the usual way
of developing native Windows applications. The developer compiles and bundles her application. The result is a directory with the executable and any dynamic link libraries needed
in it. None-international applications would deploy just this.
Then, the whole directory is imported into SDLinsight which scans the contents. All binary
files having resource sections are added to a project. The tool provides a view inspired by
Microsoft Visual Studio. The user can open all resources found and modify them in a WYSIWYG manner.

After translation of the resources, the project is regenerated in a separate directory for the
target language with all the modifications applied. The result is a localized version of the
original application (see figure 5.2).
When the application is changed the localization data can be updated.
SDLinsight supports the translation by providing a pseudo-translation feature that replaces
so far untranslated texts with mangled texts. This way a partly translated application can
be tested to get a feeling for how it looks like once it is translated. Also, proper internationalization can be verified using this functionality.
Another useful feature is the automated Quality Assurance. The software performs an audit
and presents a list of problems. Not only checks it for complete translation but also for
double spaces between words, proper punctuation, etc. The translator can then decide to fix
the issue.

Conclusion
Interestingly researching on international software was harder than expected. This is especially surprising since every computer program has to handle i18n and L10n to some extend.
Dealing with Internationalization and Localization is not an easy task. Different aspects have
to be considered depending on what view is taken. All of these aspects have to work together
when developing an international application.
Developing software for the global market is a two step process. First, culture-specific information has to be separated from the rest of the application. Here, more than just texts has
to be considered: icons, layout, time, date, currencies, etc. Also not only the user interface is
affected. As seen the application might need to handle data of different cultures. One of the
consequences is dealing with different character sets; XML technologies provide an option.
The separation of culture-specific data introduces the flexibility needed for the second step,
the localization where tools are used to support that process. The needs and skills of the
developer differ from that of a translator. Thus, available tools target different audiences.
This document provided a basic introduction on the topic which turned out to be more
complex than visible on first sight. A huge number of techniques and tools is available and
forms the basis of a small part of the industry.
