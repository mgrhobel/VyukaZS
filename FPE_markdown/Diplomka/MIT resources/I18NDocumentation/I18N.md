I18N_L10N visual tool should be implemented. It should be possible to
carry out some important steps of internationalization process in batch
process too.

\

\

Following features have to be supported:

\

**1.) Extraction of phrases from repo, see** ^**(1)**^*[**Documentation
3990: ResourceBundleRepoExport
(at.co.systema.gf.repo2.tools.ResourceBundleRepoExport)**]{.underline}*

Important improvement was made: by means of new command line parameter
-outputFileTrans can be defined the name of text file containing all
phrases

to be translated (in any case for \"none\" locale and one or more other
locales). Every \"none\" phrase is included only ONCE (clearly, no
context informations will be extracted).

If no output file exists, new one will be created. If the file already
exists, only new \"none\" phrases will be extracted (no existing
translations will be losed).

Example: running this utility against rel_3.8.3 SAP AT ECC6.0 generates
approx. 7000 phrases to be translated.

\

**2.) Extraction of phrases from all MessageBundles, ExceptionBundles
and other \*Bundle\*.properties files.**

New feature: **at.co.systema.gf.i18n.util.MessageBundlePhraseExtractor**

Investigates all \*Bundle\*.properties Files starting with \'dirRoot\'.

Write out all non duplicate reasonable phrases into some \'outputFile\'.

This could/should be the same file used for export of all Repo phrases
(see 1) by ResourceBundleRepoExport (command line parm
-outputFileTrans).

Needed improvement: new command line parameter
-overwriteExistingTranslations. If set then all already translated
phrases in the output file will be

overwriten by translated phrases from message bundles.

Example: running this utility against rel_3.8.3 SAP AT ECC6.0 generates
approx. 4000 phrases to be translated.

\

Positive side effect: check of correct message bundle keys was built in.
That means, if there is some key in Czech locale, but no key for
\"none\",

this error will be logged. So we could find out, that in
at.co.systema.MessageBundle_cs.properties many keys are in Czech.

The reason was bad import of czech phrases (\\n character iowas really

**3.) Apply some translation rules for all extracted phrases to build
the definitive phrases to be translated**

Many extracted phrases should/may not be translated.

Examples:

2a, 2b, 2c, 24x30, 2x1/2, 35x43, A (? context information needed), AB (?
context information needed), age \> 35 (dangerous, context needed),
Alter \> 35 (the same),

B, BOOLEAN, C, D, DEBUG-LEVEL, \...

\

Some rules (based on RegularExpressions) should be introduced. Every
rule has following properties:

modus: \"-\" \--\> if the translation status on phrase was
\"translate\", after succesful applying of rule, change the translation
status to \" not translate\"

\"+\" \--\> if the translation status on phrase was \"not translate\",
after succesful applying of rule, change the translation status to
\"translate\"

\"?\" \--\> do not change the phrase status, but show all phrases
fulfilling the rule.

\

batch: \"1\" \--\> this rule can be applied in automatic modus

\"0\" \--\> this rule will not be applied in automatic modus (user
interaction is needed).

\

These rules are valid for all phrases (e.g. \"A\"). This can be
overriden by context sensivite translation.

E.g. there is some repo object with I18N phrase \"A\", which is
abbreviation for \"ambulant\".

Some \"internal I18N expert\" decides, that the phrase \"A\" should be
translated to \"X\" for this repo object.

This context information should be saved in some way.

\

If there is only one context information for some phrase (mostly the
case by MessageBundle phrases), this could be exported

into the \"translation file\" in some special column \"context info\".

For other phrases like \"Aufenthalt\", \"Patient\", \"Anforderung\",
etc. no context information can be exported into the \"translation
file\",

because there are many repo objects containing these phrases.

**4.) Import of phrases from translation file into the repository**

New implementation of ResourceBundleRepoImport
(ResourceBundleRepoImport2) should be implemented.

Concept: All I18N objects will be examined (exactly like by
ResourceBundleRepoExport). For every object its

\"none\" phrase will be searched in the \"translation file\". If found,
existing translations for every locale from \"translation file\"

should be imported into the repository. If needed, new I18N object for
corresponding locale should be created and saved.

Context Rules should be applied in order to keep the context dependent
translation (if any).

\

**5.) Import of phrases from translation file into MessageBundle
Files.**

To be implemented:
**at.co.systema.gf.i18n.util.MessageBundlePhraseImporter**

It investigates all \*Bundle.properties files (that means for \"none\"
locale). For every key the corresponding phrase will be

searched in the translation file. For all translated locales the
corresponding translations will be saved in the corresponding

\*Bundle_locale.properties.

\

Carry out some check of formatting: That means the number of symbols
\<CR\>, \<TAB\>, {0}**,** {1}**, \...** must be the same in the orginal
phrase as in the translated phrase.

**8.) Localized names of MetaRecords (LogObjects), Relations and
Columns**

Utility **at.co.systema.gf.dict.tools.LogObjectResourceBundleCreator**
generates MessageBundle keys for

all metarecords from repository.

Bundle at.co.systema.gf.dict.MessageBundle.properties (and bundles for
all allowable locales) get new key

LogObjectName.key=\...

e.g.

OrgEinheit.key=Organisationseinheit

By command line parameters -relations and/or -columns the keys for
relations and/or columns will be generated too:

LogObjectName.RelationName.key=

LogObjectName.ColumnName.key=

\

Some optimizations were implemented here, e.g. no metarecord_search keys
will be generated, because there is already

metarecord.key generated. Ths reduces the number of generated keys.

By generating of column keys for metarecord XYZ and metarecord
XYZ_search every column should be \"extracted\" only once.

between possible translations**

My imagination: Small checkbox (default = yes) by every phrase, if the

General issue:

For every release (patches must be taken into account too) there is one
file \"mpa_phrases_rel_3.11.xls\" checked in ????

This file will be generated in points 1 and 2.

Mechanism for automatical processing of translation relevant patches
must be introduced.

Phrases from 2) are release dependant

Phrases from 1) are repo dependant (syex customizing \--\> vendor
dependent).

That means that per release the extraction from repo should be done for
every vendor to get all possible phrases

supported by MPA on syex und syst level (or somebody has to decide what
is the reference repository for basic MPA translation).

This process should be done manually.

\

Some special treatment for all phrases, they are \"correct HTML
snippets\":

\<font size=5 color=red\>Darf im Repo nicht verändert werden!\</font\>

\<html\> \<p\>\<font color=\"#FF0000\"\>\<font face=\"Arial Narrow\"
size=\"7\"\>Bitte melden sie sich am Aufnahmeschalter.\</font\>\</font\>
\</p\> \</html\>

\

Only HTML output string should be extracted.

By Import of such phrases must be the same logic inverted.

\

Treating of \"HotKeys = Mnemonics\" in MessageBundles like:

LogonPanel.Ok=&Ok

LogonPanel.Cancel=&Abbrechen

LogonPanel.Beenden=&Beenden

in at/co/systema/MessageBundle.properties

We have 2 possibilities here:

a\) we let the values so as they are and the translator should learn
about the purpose of \"&\" sign.

Anyway, he does no see these 3 phrases sequentially, because he sees all
phrases alphabetically and can not \"judge\" the right unambiguous use
of \"&\" signs in one dialog.

b\) we extract only the phrase without \"&\" and hotkey \"O\" in the
additional column of table i18n_trans.
